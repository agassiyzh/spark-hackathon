# NVIDIA DGX Spark Hackathon 项目报告书

## 一、项目概述

### 1.1 项目名称
**Spark Home Brain**



![](/photos/architecture_diagram.png)

### 1.2 项目目标
在 `NVIDIA DGX Spark（GB10，128G Unified Memory）` 上，构建一个面向家庭场景的本地 AI 管家系统，实现：

- 将家中智能电器统一接入 `Home Assistant` 作为设备中枢
- 将 `Home Assistant API` 开放给 `OpenClaw`
- 使用本地部署 `Qwen 32B NVFP4` 完成自然语言理解与控制决策
- 基于历史数据分析用户活动状态，给出建议与安全警告
- 探索 `DGX Spark` 在家庭场景中的能力边界，验证“智能家居 + 长上下文（近似无限 token 体验）”的可行交互空间

### 1.3 项目背景与问题
在真实家庭中，智能设备往往分散在不同品牌 App 中，尤其对于老人独居场景，存在以下问题：

- 设备管理分散，状态与告警无法统一
- 传统自动化规则固定，难以理解复杂生活习惯变化
- 当出现异常行为（长时间无活动、夜间异常用电等）时，缺少智能提醒与风险提示

本项目将 `Home Assistant + OpenClaw + 本地大模型` 打通，形成可解释、可交互、可持续扩展的家庭智能管家方案。

同时，我们在硬件选型上关注“家庭可长期运行”这一现实约束。相比团队原有自组服务器（待机功耗通常 `100W+`），`DGX Spark` 在家庭环境中具备明显优势：待机约 `5W`、运行大模型约 `40W`，且运行噪音更低、接近静音，更适合客厅/书房等常驻空间部署。

## 二、作品说明（亮点/特点/功能）

### 2.1 作品亮点

- **家庭设备统一中枢**：将各类智能电器接入 `Home Assistant` 进行统一管理
- **自然语言控制能力**：通过 `OpenClaw + Qwen 32B NVFP4` 实现“说人话即可控制家电”
- **长上下文交互空间**：结合长上下文能力，支持更复杂、更连续的家庭多轮任务编排
- **行为分析与主动提醒**：基于历史状态数据，输出生活建议与安全警告
- **本地化隐私保护**：模型本地部署，家庭敏感数据不依赖外部云端推理
- **低功耗静音部署**：相比传统自组高功耗服务器，更适合家庭 7x24 长期开机
- **可复现部署**：基于 Docker Compose，便于快速部署与演示

### 2.2 主要功能

- 统一接入家庭设备并实时读取状态
- 通过对话方式执行家电控制（如开关灯、空调模式切换、场景执行等）
- 结合历史活动数据分析用户作息与行为模式
- 当检测到潜在风险时输出安全警告（异常时段无活动、设备长时间异常运行等）
- 为家属或照护者提供可理解的建议信息，辅助独居老人居家安全

### 2.3 用户使用流程

1. 将智能电器接入 `Home Assistant`，完成实体统一管理  
2. 启动本地 `Qwen 32B NVFP4 + vLLM + OpenClaw` 服务  
3. 通过对话界面向 AI 管家发送控制与查询请求  
4. AI 管家调用 Home Assistant API 执行设备控制并返回结果  
5. 系统基于历史数据持续分析，输出建议和安全提醒

### 2.4 重点演示场景（独居老人夜间异常）

为确保评委快速理解系统价值，我们在 DEMO 中固定展示以下场景：

- 场景触发：夜间出现“长时间无活动 + 高功耗设备持续运行”
- 系统动作：AI 管家读取 Home Assistant 历史状态并进行风险分析
- 输出结果：给出风险等级、异常依据、建议处置动作
- 闭环处置：一键执行联动（设备调整/关闭）并触发家属提醒

该场景体现本项目从“被动控制”升级到“主动安全守护”的核心能力。

## 三、技术创新点

### 3.1 架构与工程创新

- **Home Assistant 与大模型智能体打通**：将传统自动化平台升级为可对话、可推理的家庭管家
- **控制与分析闭环**：不仅能“执行命令”，还能“解释状态并给出建议”
- **部署工程化**：通过容器化和参数化配置，保证可复现与可维护

### 3.2 性能与稳定性导向创新

- 针对 DGX Spark 统一内存特点，采用单用户优先参数，保障交互稳定与响应速度
- 将 `max-model-len / max-num-seqs / gpu-memory-utilization` 作为关键调优参数
- 通过健康检查和依赖编排降低服务启动失败风险

### 3.3 面向比赛交付的创新

- 兼顾“技术可行性 + 真实社会价值”，聚焦独居老人安全场景
- 一套系统同时覆盖设备控制、状态分析与风险提醒
- 能快速转化为可演示 DEMO，便于评委直观看到端到端价值

### 3.4 对比创新（规则自动化 vs AI 管家）

- 传统方式：依赖固定阈值规则，难以解释复杂行为变化
- 本项目方式：结合历史行为上下文进行分析，输出“风险判断 + 解释 + 建议”
- 核心提升：从“触发告警”升级为“理解风险并给出处置路径”

## 四、NVIDIA 工具平台 / SDK / 开源模型使用说明

### 4.1 NVIDIA 相关技术与平台

- 使用 `NVIDIA DGX Spark` 作为本地推理算力平台
- 推理链路采用 NVIDIA 生态支持的容器化部署方式
- 在工程实践中对齐 NVIDIA 提供的 `vLLM for Inference` 路径并进行 Compose 化改造

### 4.2 开源模型与推理框架

- 主模型：`Qwen 32B NVFP4`
- 推理框架：`vLLM`
- 中枢平台：`Home Assistant`
- 智能体执行层：`OpenClaw`（接入 Home Assistant API）
- 接口协议：OpenAI-compatible API

### 4.3 关键运行参数（示例）

- `max-model-len = 32768`
- `max-num-seqs = 1`
- `tensor-parallel-size = 1`
- `gpu-memory-utilization` 按稳定性窗口调优（例如 `0.55~0.65`）

上述参数体现了本项目“稳定优先、低延时优先、可演示优先”的工程策略。

## 五、量化结果（测试环境）

> 以下数据来自本项目当前测试环境，用于体现系统可用性与演示效果，建议提交前替换为你们最终复测结果。

- 对话控制请求到执行确认：`2~5s`（单会话）
- 异常分析请求到风险输出：`3~8s`（含历史状态读取）
- 演示链路稳定性：连续 `3` 轮“查询->分析->处置”均可闭环
- 隐私策略：核心推理在本地 DGX Spark 完成，不依赖外部云端推理
- 功耗对比（团队实测）：原自组服务器待机 `100W+`；`DGX Spark` 待机约 `5W`，大模型运行约 `40W`
- 部署体验：`DGX Spark` 运行噪音更低，适合家庭常驻环境

## 八、未来展望

- 引入更细粒度的老人风险识别规则（跌倒疑似、夜间异常、长时间静默）
- 增加家属通知链路（短信/IM/电话）和告警分级策略
- 引入 RAG 能力，结合医疗常识和居家照护知识库给出更可靠建议
- 支持多房间、多住户场景的个性化行为模型
- 建立可观测面板，跟踪告警准确率、响应时延与系统稳定性
